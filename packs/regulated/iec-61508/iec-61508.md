# IEC 61508 (E/E/PE functional safety) — wf compliance checklist

**NOT A COMPLIANCE TOOL.** This pack provides structured, agent-facing
reminders only. It does NOT certify or demonstrate conformance with any
safety or security standard. Engage an accredited assessor before claiming
compliance. Evidence captured here is a starting point, not a substitute
for a certified V&V process.

  Purchase: IEC Webstore (iec.ch).
- ISO/IEC/IEEE 12207:2017 — Software lifecycle processes (companion

- ISO/IEC 25010:2023 §4.2.9 — Safety quality characteristic.
- Notified body list: ec.europa.eu/growth/tools-databases/nando

## Severity / criticality classification

Safety Integrity Levels: SIL 1 (low risk) → SIL 4 (high risk,
nuclear/aerospace adjacent). Assign the SIL at system level at Frame via
risk assessment; software inherits or is allocated a SIL.
**An undetermined SIL is a blocking ambiguity, not a detail.**

## Per-phase obligations (wf phases)

- **Frame**: system safety function identified; SIL determined via risk
  assessment (61508 Part 3 §7.4) and recorded (requirement/ADR record).
  Safety Requirements Specification captured — each safety requirement a
  `requirement` record with verifiable ACs, traceable to a hazard.
- **Context**: existing safety architecture mapped in the context map;
  independence between safety-critical and non-safety code paths
  confirmed; requirements baseline approved (scope approval).
- **Design**: 61508 Part 3 design techniques applied for the target SIL
  (structured programming, defensive coding, fault detection); ADR
  required for each technique choice; testability sketch per AC.
- **Plan**: coding standard selected, documented, and approved (MISRA C /
  MISRA C++); verification strategy names the structural coverage target
  per SIL.
- **Build**: selected coding standard applied; tests before production
  code; no dynamic memory allocation on safety paths (SIL 3/4) holds in
  the diff.
- **Verify**: structural coverage per SIL — SIL 1: statement; SIL 2:
  branch; SIL 3/4: MC/DC — recorded as grounded metric records; static
  analysis mandatory; coverage results in the test report.
- **Ship**: evidence package compiled (below); tool qualification records
  confirmed for any tool that produces safety evidence; release dossier
  assessor-ready before deployment; retro reviews SIL assumptions.

## Evidence package contents

- Safety Requirements Specification (SRS) with hazard-to-requirement traceability
- Software Architecture Description with safety measure rationale
- Coding standard selection and compliance report
- Unit / integration test report with structural coverage metrics
- Static analysis report with documented deviations
- Tool qualification records for safety-relevant tools
- Software Configuration Management (SCM) audit log
- Assessor-facing release dossier checklist
