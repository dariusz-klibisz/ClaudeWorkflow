# ISO 26262 (automotive functional safety) — wf compliance checklist

**NOT A COMPLIANCE TOOL.** This pack provides structured, agent-facing
reminders only. It does NOT certify or demonstrate conformance with any
safety or security standard. Engage an accredited assessor before claiming
compliance. Evidence captured here is a starting point, not a substitute
for a certified V&V process.

## Authoritative sources

- ISO 26262:2018 — Functional Safety for Road Vehicles (all 12 parts);
  Part 6 (Software) and Part 8 (Supporting processes) most relevant.
  Purchase: ISO Store (iso.org).
- Automotive SPICE (A-SPICE) v4.0 — process assessment profile
  (automotivespice.com).
- AUTOSAR Classic/Adaptive — architecture standards (cross-reference only).
- ISO/IEC/IEEE 12207:2017 — lifecycle companion.

## Severity / criticality classification

Automotive Safety Integrity Levels: QM (no safety requirement) → ASIL A →
B → C → D (highest). Assign the ASIL at Frame from a Hazard Analysis and
Risk Assessment (HARA); ASIL decomposition is permitted for redundant
paths. **An unassigned ASIL is a blocking ambiguity, not a detail.**

## Per-phase obligations (wf phases)

- **Frame**: item definition and boundaries; HARA performed → ASIL
  recorded (requirement/ADR record). Software Safety Requirements (SSR)
  traceable to system safety goals — each SSR a `requirement` record with
  verifiable ACs.
- **Context**: safety mechanisms the architecture must carry (error
  detection, monitoring) identified in the context map; A-SPICE process
  level target confirmed; SSR baseline approved (scope approval).
- **Design**: design notations per 26262-6 §7.4 (e.g., semi-formal
  notations/state machines for ASIL C/D); safety mechanisms placed;
  architecture ADR authored; testability sketch per AC.
- **Plan**: MISRA C:2012 compliance plan (mandatory ASIL B+); ASIL C/D: no
  recursion, bounded loops; ASIL B+: no dynamic memory allocation;
  verification strategy names the coverage target per ASIL.
- **Build**: coding standard applied (cite MISRA rule IDs in reviews);
  tests before production code; the ASIL B+ constraints hold in the diff.
- **Verify**: coverage per ASIL — A: statement; B: branch; C/D: MC/DC —
  recorded as grounded metric records; back-to-back testing for safety
  mechanisms; results per SSR in the test report.
- **Ship**: evidence package compiled (below); tool confidence level (TCL)
  and tool qualification confirmed for all safety-related tools;
  confirmation-measures checklist (26262-8 §9); retro reviews HARA
  assumptions against field data.

## Evidence package contents

- Item Definition and HARA report
- Software Safety Requirements (SSR) with ASIL allocation
- Software Architectural Design documentation
- Coding guideline compliance report (MISRA)
- Unit / integration / system test report with coverage metrics per ASIL
- Tool confidence level assessments and qualification records
- Safety case / Confirmation Measure checklist (26262-8 §9)
- A-SPICE process assessment evidence (if contractually required)
