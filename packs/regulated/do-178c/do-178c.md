# DO-178C (airborne software) — wf compliance checklist

**NOT A COMPLIANCE TOOL.** This pack provides structured, agent-facing
reminders only. It does NOT certify or demonstrate conformance with any
safety or security standard. Engage an accredited assessor before claiming
compliance. Evidence captured here is a starting point, not a substitute
for a certified V&V process.

  Certification (RTCA/EUROCAE ED-12C). Purchase: rtca.org or eurocae.net.
- DO-330 — Software Tool Qualification Considerations (companion to
  DO-178C).
- DO-331, DO-332, DO-333 — supplements for Model-Based Development, OO&RT,
  Formal Methods.
- ARP4754A — Guidelines for Development of Civil Aircraft and Systems
  (system-level companion).
- FAA AC 20-115D (US) and EASA CM-SWCEH-001 (EU) — regulatory acceptance
  references.

## Severity / criticality classification

Design Assurance Levels: DAL E (no safety effect) → D → C → B → A
(catastrophic failure effect). Assign the DAL at Frame from the
system-level FHA/PSSA; software inherits or is allocated a DAL.
**An undetermined DAL is a blocking ambiguity, not a detail.**

## Per-phase obligations (wf phases)

- **Frame**: DAL determined from the Functional Hazard Assessment and
  recorded (requirement/ADR record); Plan for Software Aspects of
  Certification (PSAC) established. Software Requirements (high-level in
  DO-178C terms) captured — each a `requirement` record with verifiable
  ACs, linked to system requirements in the RTM; non-verifiable
  requirements remaining is a blocking ambiguity.
- **Context**: the four DO-178C plans established — Software Development
  Plan, Verification Plan, Configuration Management Plan, and Quality
  Assurance Plan; requirements baseline approved (scope approval); tool
  qualification needs confirmed.
- **Design**: Software Architecture and Low-Level Requirements produced;
  ADR for partitioning strategy; for DAL A/B: formal design review;
  testability sketch per AC.
- **Plan**: compliance with DO-178C §6 planned (no dead code, no
  deactivated code in the released build); source code reviews required
  for DAL A/B; verification strategy names the coverage target per DAL.
- **Build**: coding standards applied; tests before production code; no
  dead/deactivated code in the released build holds in the diff.
- **Verify**: structural coverage per DAL — C: statement; B: decision;
  A: MC/DC — recorded as grounded metric records; requirement-based tests
  trace to every LLR; coverage data in the test report.
- **Ship**: Software Accomplishment Summary (SAS) compiled; all DO-178C
  lifecycle data and evidence confirmed complete and correct; SAS
  submitted to DER/ACO (FAA) or Design Organisation (EASA) —
  certification credit granted only after authority review; retro feeds
  service difficulty reports into SAS updates.

## Evidence package contents

- Plans: PSAC, Software Development Plan, Verification Plan, CM Plan, QA Plan
- Software Requirements Document (high-level + low-level)
- Requirements traceability matrix (HLR ↔ LLR ↔ Source ↔ Test)
- Source code with no dead code (documented and justified exceptions)
- Structural coverage report (per DAL: statement / decision / MC/DC)
- Tool qualification records (DO-330) for all tools generating certification credit
- Problem reports and configuration index
- Software Accomplishment Summary (SAS) — the primary certification deliverable
