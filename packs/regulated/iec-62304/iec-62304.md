# IEC 62304 (medical device software) — wf compliance checklist

**NOT A COMPLIANCE TOOL.** This pack provides structured, agent-facing
reminders only. It does NOT certify or demonstrate conformance with any
safety or security standard. Engage an accredited assessor before claiming
compliance. Evidence captured here is a starting point, not a substitute
for a certified V&V process.

## Severity / criticality classification

Software Safety Classes: Class A (no injury possible) → Class B
(non-serious injury possible) → Class C (serious injury or death
possible). Assign at Frame; drives verification rigor throughout.
**An unassigned safety class is a blocking ambiguity, not a detail.**

## Per-phase obligations (wf phases)

- **Frame**: Software Safety Classification determined (62304 §4.3) and
  recorded (requirement/ADR record); intended use and reasonably
  foreseeable misuse identified. Software Requirements Specification
  traceable to device-level requirements — each requirement a
  `requirement` record with verifiable ACs, linked to hazard-control
  measures (14971 §9); RTM column: SRS ↔ risk control.
- **Context**: software system architecture defined in the context map;
  SOUP identified — listed in a SOUP inventory with version and known
  anomalies; requirements baseline approved (scope approval).
- **Design**: Software Architectural Design per 62304 §5.4; for Class C:
  formal inspections of design documentation; architecture ADR authored;
  testability sketch per AC.
- **Plan**: coding standards documented (62304 §5.5); new SOUP evaluated
  for suitability; for Class C: mandatory code reviews planned;
  verification strategy approved.
- **Build**: coding standards applied; tests before production code for
  Class B/C; for Class C: code reviews conducted during implementation.
- **Verify**: unit testing required for Class B/C; system testing against
  all software requirements; regression testing when requirements change;
  pass/fail per requirement recorded as grounded metric records in the
  test report.
- **Ship**: evidence package compiled (below); change control process
  (62304 §6) confirmed followed for all modifications; SOUP inventory
  updated; release requires completed risk management file (14971) and
  software release checklist with QMS sign-off (13485); retro feeds
  post-market surveillance findings back into the risk management file.

## Evidence package contents

- Software Safety Classification rationale
- Software Requirements Specification with traceability to device requirements
- SOUP inventory (name, version, manufacturer, known anomalies, evaluation summary)
- Software Architectural Design documentation
- Unit and system test reports with requirement traceability
- Risk management file (14971) with software risk contributions
- Problem resolution records (62304 §9)
- Software release checklist with QMS and regulatory body sign-off fields
