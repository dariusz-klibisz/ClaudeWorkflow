# NIST SP 800-53 (US federal information systems) — wf compliance checklist

**NOT A COMPLIANCE TOOL.** This pack provides structured, agent-facing
reminders only. It does NOT certify or demonstrate conformance with any
safety or security standard. Engage an accredited assessor before claiming
compliance. Evidence captured here is a starting point, not a substitute
for a certified V&V process.

## Authoritative sources

- NIST SP 800-53 Rev. 5 — Security and Privacy Controls for Information
  Systems and Organizations. Free download: nvlpubs.nist.gov
  (csrc.nist.gov/publications).
- NIST SP 800-218 v1.1 (SSDF) — Secure Software Development Framework.
  Free: csrc.nist.gov.
- CMMI v3.0 — Capability Maturity Model Integration.
  appraisals.cmmiinstitute.com.
- FedRAMP — cloud-service authorization framework. fedramp.gov
  (cross-ref; applies when hosting US federal data).
- EO 14028 — Executive Order on Improving the Nation's Cybersecurity
  (legally binding for federal contractors).

## Severity / criticality classification

FIPS 199 impact levels: Low → Moderate → High (based on Confidentiality,
Integrity, Availability). Determine the system categorisation at Frame;
NIST SP 800-60 Vol. 2 provides category assignments by information type.
**An unagreed categorisation — or unselected control baseline — is a
blocking ambiguity, not a detail.**

## Per-phase obligations (wf phases)

- **Frame**: system classified using FIPS 199 and recorded
  (requirement/ADR record); applicable overlays identified (Privacy,
  FISMA, CUI); CMMI target level determined (contractually required
  level, typically ML 2 or ML 3). Security and privacy requirements
  captured traceable to 800-53 control families — each a `requirement`
  record with verifiable ACs; RTM: requirement ↔ control ↔ test case;
  control baseline (Low/Moderate/High) selected.
- **Context**: existing System Security Plan (SSP) reviewed; inherited
  controls identified in the context map; Plan of Action and Milestones
  (POA&M) established for known gaps; requirements baseline approved
  (scope approval).
- **Design**: SSDF practices applied — PO.1 (define security
  requirements), PS.1 (protect code), PW.1 (design for security); ADR
  for each significant security architecture decision; threat modelling
  (STRIDE or equivalent); testability sketch per AC.
- **Plan**: SSDF PW.4 (review for vulnerabilities) and PW.5 (reuse vetted
  software) planned; supply chain risk management — vet new dependencies
  against NVD; no hardcoded credentials; verification strategy approved.
- **Build**: secure coding practices applied; tests before production
  code; no hardcoded credentials in the diff.
- **Verify**: SSDF PW.7 (test for vulnerabilities); SAST and DAST results
  documented as grounded metric records; penetration test for High-impact
  systems; test results mapped back to 800-53 controls.
- **Ship**: SSDF RV.1 (identify vulnerabilities) and RV.2 (assess); SBOM
  produced (EO 14028 requirement); POA&M generated or updated; ATO
  evidence package prepared and submitted to the Authorising Official
  (AO) — for cloud: FedRAMP authorization package; CMMI appraisal
  evidence collected if required by contract; retro updates the
  continuous monitoring plan and captures incident response after-action
  reviews.

## Evidence package contents

- System Security Plan (SSP) with 800-53 control implementation statements
- FIPS 199 system categorisation and information type identification
- Threat model report (STRIDE or PASTA)
- SAST, DAST, and penetration test reports with remediation tracking
- SBOM (SPDX or CycloneDX format — both approved under EO 14028)
- Plan of Action and Milestones (POA&M)
- Supply chain risk assessment for key dependencies
- ATO package (or FedRAMP authorization package for cloud)
- CMMI appraisal evidence (if contractually required)
