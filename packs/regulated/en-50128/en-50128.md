# EN 50128 (railway software) — wf compliance checklist

**NOT A COMPLIANCE TOOL.** This pack provides structured, agent-facing
reminders only. It does NOT certify or demonstrate conformance with any
safety or security standard. Engage an accredited assessor before claiming
compliance. Evidence captured here is a starting point, not a substitute
for a certified V&V process.

  Control and Protection Systems. Purchase: cenelec.eu or national
  standards bodies.
- EN 50657:2017 — Railway Applications: Software for Rolling Stock
  Control Systems (onboard).
- EN 50129:2018 — Railway Applications: Safety-related Electronic Systems
  for Signalling (cross-ref).
- EN 50126 parts 1+2 — RAMS framework (cross-ref; system-level).

## Severity / criticality classification

Software Integrity Levels: SIL 0 (no safety requirement) → SIL 1 → SIL 2
→ SIL 3 → SIL 4 (highest). Assign the SIL from system RAMS analysis at
Frame. EN 50128 mandates independence requirements per SIL.
**An unassigned SIL — or undefined independence roles — is a blocking
ambiguity, not a detail.**

## Per-phase obligations (wf phases)

- **Frame**: SIL determined from EN 50126 RAMS analysis and recorded
  (requirement/ADR record); required independence level identified
  (T1/T2/T3 roles for SIL 3/4). Software Requirements Specification
  captured — complete, consistent, and verifiable; each a `requirement`
  record with verifiable ACs; RTM from system requirements to software
  requirements; non-verifiable requirements remaining is a blocking
  ambiguity.
- **Context**: Software Quality Assurance Plan established; Validator
  role (independent from Developer for SIL 2+) identified; requirements
  baseline approved (scope approval).
- **Design**: Software Architecture Document produced; EN 50128 §5.3
  mandates Defensive Programming techniques for SIL 2+; ADR for each
  safety mechanism; testability sketch per AC.
- **Plan**: coding standard selected and documented; SIL 3/4: no dynamic
  memory, no implicit type conversion, bounded loops; code review by
  independent T2/T3 planned; verification strategy approved.
- **Build**: coding standard applied; tests before production code; code
  review by independent T2/T3 for SIL 3/4; the SIL 3/4 constraints hold
  in the diff.
- **Verify**: structural coverage per SIL — SIL 1: statement; SIL 2:
  branch; SIL 3/4: MC/DC — recorded as grounded metric records;
  independence of test team required for SIL 2+; back-to-back simulation
  testing for SIL 3/4.
- **Ship**: evidence compiled for Validator review; all software
  lifecycle documents cross-reference correctly; configuration audit
  done; independent Validator signs off the Software Validation Report;
  National Safety Authority or notified body approval may be required for
  traffic-critical applications; retro feeds incidents and near-misses
  back into SIL assumptions.

## Evidence package contents

- Software Requirements Specification with traceability to system RAMS requirements
- Software Architecture Document with safety mechanism rationale
- Coding standard selection and compliance evidence
- Test reports with structural coverage metrics and independence records
- Software Quality Assurance Plan and audit results
- Independent Validator review records and Software Validation Report
- Software Configuration Index and change log
- Notified body / National Safety Authority correspondence (if applicable)
