# Workflow Redesign — Claude-Native Development Workflow

> Design documentation for a ground-up, **Claude Code-only** redesign of the
> development workflow currently generated by `init_claude.py` (v0.36).
> Grounded in (a) the full runtime analysis of the current scaffold
> ([docs/workflow-reference/](../workflow-reference/README.md)) including its
> 51-item issues catalog, and (b) deep verification research against the current
> Claude Code documentation (all mechanisms cited, with minimum versions).
>
> Status: **implemented** — the design was built (M0–M5 delivered; see
> [09-roadmap.md](09-roadmap.md)). These documents remain the design record
> and rationale; the engine and plugin in this repo are the implementation.

## Goals (from requirements)

1. **Claude Code only.** Design around native mechanisms (plugins, hooks, skills,
   subagents, tasks) — no cross-tool lowest-common-denominator layer.
2. **No skipping steps.** Every "must" in the workflow maps to a *blocking*
   mechanism, not prose. An agent that "thought it was done" or "didn't do
   something" is stopped by the tooling, not by its own discipline.
3. **Compaction-proof.** The workflow survives context compaction without losing
   phases, steps, documents, or obligations. On-disk state is the only source of
   truth; context is re-anchored automatically.
4. **Easy to update.** Installing the workflow into a project is one settings
   entry; updating it is one plugin update. No migrations, no re-generation, no
   per-project template drift.
5. **Reference-corpora-backed.** The `Design`, `Coding` (and optionally `UI_UX`)
   best-practice libraries ship inside the plugin and are routed per agent/file.

## Relationship to prior work

- `docs/workflow-reference/` documents the **as-is** system (v0.36) and its
  issues catalog (A1–G6). This redesign treats that catalog as requirements:
  [04-enforcement-spine.md §6](04-enforcement-spine.md) cross-references every
  item to the mechanism that makes it impossible.
- The current generator (`claudeinit/`) is **not evolved** — it is replaced. The
  proven concepts it pioneered (evidence ledger, fail-closed gates, no-wedge
  invariant, loop records, lessons lifecycle, clean-template discipline) are
  carried forward; its delivery model (emit + migrate) and its enforcement
  implementation (bash+python heredocs parsing prose) are not.

## Founding rules (lessons paid for by v0.13–v0.36)

1. **Verify facts, not prose.** A gate reads structured records the engine
   wrote, never scans free text the agent wrote (the v0.30 lesson,
   generalized to 100%).
2. **Single source for every contract.** One constant, one parser, one roster —
   consumers are generated, never hand-mirrored (the phase_gate.sh /
   GATING_AGENT_NAMES lessons).
3. **No wedge, full audit.** Every gate has an escape (`park`, `force`) that
   works even when the gate itself is broken, and every escape is a recorded,
   surfaced event.
4. **Honest bounds stay honest.** What cannot be machine-verified (human
   approval) is recorded and *flagged*, never faked.
5. **Enforce at the earliest event that can block.** Task completion blocks at
   `TaskCompleted`, reviewer verdicts at `SubagentStop`, phase exit at the
   engine gate, turn end at `Stop` — not "later, at Trace".
6. **The checklist is always in context.** Not "the agent should read the step
   file" — the engine injects the current contract into every turn.
